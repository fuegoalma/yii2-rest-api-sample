# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Yii2 (basic template) REST API for users, albums, and photos. PHP 8.5, MySQL 8, everything runs inside Docker (Compose **v2** — invoke it as `docker compose`, with a space, not the legacy `docker-compose`) — all PHP/console commands must be executed through `docker compose exec web` (use the `Makefile` shortcuts below, e.g. `make test` instead of typing the full command). The environment is defined by a single multi-stage `Dockerfile` (`base` → `dev` → `prod`); the **Imagick** extension (used for photo uploads) plus `pdo_mysql`/`mysqli`/`pcntl` (the queue worker's graceful shutdown) and Composer are baked into the `base` stage. `docker-compose.yml` builds the `dev` stage (`target: dev`) and bind-mounts the project directory, so `composer install` and the code live on the host and container startup is instant; the CD pipeline builds the self-contained `prod` stage. Keep the `Dockerfile`'s `base` stage as the single source of truth for the runtime — don't reintroduce runtime extension installs in compose. Alongside `web` there is a **`cron`** service that reuses the same image (shared `image: yii-app:dev` name → built once) but runs the cron daemon instead of Apache: its entrypoint (`docker/cron/entrypoint.sh`) snapshots the container env into `/etc/container-env` (cron jobs run with a bare environment) and installs the versioned schedule `docker/cron/crontab` into `/etc/cron.d/app-cron`. That crontab is the single place to declare scheduled console commands — currently a daily `refresh-token/prune`; each line sources the env and logs to the container stdout (`docker compose logs cron`). There is also a **`worker`** service (same shared image) whose entrypoint is the long-running queue worker `yii queue/listen` instead of Apache/cron, with `restart: unless-stopped` — see **Background jobs (queue)**. `make rebuild` builds the image and (re)starts `web`, `cron` and `worker`.

## Codebase Navigation (CodeGraph)

This repo has a [CodeGraph](https://github.com/colbymchenry/codegraph) index (`.codegraph/`, local-only, gitignored). **Prefer it over `grep`/`find`/reading whole files** when the question is "where is X" / "who calls X" / "what does X touch" — it answers directly from the index instead of spending tokens scanning file contents:

- `codegraph query "<name>"` — find a symbol (class/method/interface/file) by name.
- `codegraph explore "<question>"` — relevant symbols' source + call paths for an area, in one shot (e.g. `codegraph explore "how are refresh tokens rotated"`).
- `codegraph node <name>` — one symbol's source plus its callers/callees.
- `codegraph callers <symbol>` / `codegraph callees <symbol>` — who calls a method / what it calls.
- `codegraph impact <symbol>` — what else would be affected by changing a symbol (use before refactoring a shared class like `BaseCrudService`, `ApiController`, or a contract in `models/contract/`).
- `codegraph affected <files...>` — which test files cover a set of changed source files.

Run `codegraph sync` after a batch of edits (or `codegraph status` to check if the index is stale) so later lookups reflect the current code. Still use `Read`/`Edit` for the actual file content once you know where to look — CodeGraph replaces the *searching*, not the reading.

**All code MUST follow SOLID, DRY, and KISS.** Non-negotiable for every change: no duplicated logic (extract shared code into base classes/traits/helpers, including tests), depend on interfaces from `models/contract/` rather than concretions, keep each class to a single responsibility, and prefer the simplest design that works — don't add abstractions for hypothetical future needs. When touching existing code that violates these principles, fix the violation rather than extending it.

## Commands

All PHP/console commands run inside the `web` container; use the `Makefile` shortcuts (`make help` lists all of them) instead of typing `docker compose exec web ...` by hand:

```bash
# First-time setup
./init.sh          # creates .env from .env.example
./setup.sh         # starts Docker, composer install, creates main + test DBs, runs migrations on both

# Migrations (two DBs: main and test — keep both migrated)
make migrate                        # both DBs
make migrate-main                   # main only
make migrate-test                   # test only

# Generate migrations from existing DB schema (bizley/yii2-migration)
make migration-create table=<table>   # or table='*' for all
make migration-update table=<table>   # diff against migration history

# Seed data
make seed                # optional count=N arg, default 10
make seed-clear

# Tests (Codeception, run against TEST_DB_NAME via config/test.php)
make test                                            # all
make test-functional
make test-unit
make test-one suite=functional class=UsersCest                 # one class
make test-one suite=functional class=UsersCest:testMethodName  # one test
make build                # after changing Codeception modules

# Code style (PSR-12, PHP CS Fixer)
make cs-check             # dry-run, shows violations/diff
make cs-fix               # auto-fix

# Static analysis (PHPStan, level 5, config in phpstan.neon.dist)
make stan
```

The `Makefile` wraps `docker compose exec -T web ...` (see `make sh` for an interactive shell into the container, which keeps a TTY). Adding a new recurring command should get a `Makefile` target rather than being typed out in full each time.

App is served at http://localhost:8084, phpMyAdmin at http://localhost:8085, MySQL exposed on host port 3307. DB credentials come from `.env` (read via `getenv()` in `config/db.php` / `config/test_db.php`).

## Architecture

Layered flow for every endpoint: **Controller → Form Request → Service → Repository → ActiveRecord model**, with interfaces in `models/contract/` defining the service and repository shapes.

- `controllers/basic/ApiController.php` — abstract base extending `yii\rest\ActiveController`. It sets up CORS/JSON + JWT bearer auth (shared `controllers/basic/ApiControllerTrait.php`), replaces the default REST actions with its own generic `actionIndex/View/Create/Update/Delete` implementations that delegate to the constructor-injected `$service` (an `ApiServiceInterface`) and gate every action through the injected `AccessControlInterface` `$access` (see **Authorization (RBAC)**). Concrete controllers set `$modelClass`, implement the create/update/search forms, and implement `accessResource()` (the permission-name prefix, e.g. `'album'`). They override `requireCollectionAccess()`/`requireMemberAccess()` when their permissions don't follow the `<resource>.<action>[.any]` naming (e.g. `RolesController`), and `assertVisible()` for resource-specific visibility (a soft-deleted album is a 404 outside the review audience — `AlbumsController`/`PhotosController` via `AlbumVisibilityTrait`). Override an action only when the response differs (e.g. `AlbumsController::actionView` returns an `AlbumViewResponse` DTO).
- `PhotosController` — photos are a **child resource of albums**, so listing and creation are nested (`GET|POST /albums/<albumId>/photos`) while member actions stay flat (`GET|PUT|DELETE /photos/<id>`); there is deliberately no flat `/photos` listing. It overrides `actionIndex(int $albumId)` (reuses `handleIndex()` but pins the album scope via `PhotoService::getByAlbum($albumId, $criteria)`) and `actionCreate(int $albumId)` (grabs the uploaded file via `UploadedFile::getInstanceByName('file')`, then reuses `handleWrite()` with `PhotoService::createInAlbum()`); `view`/`update`/`delete` use the generic base actions. `PhotoService` implements `PhotoServiceInterface` (extends `ApiServiceInterface`) and additionally depends on `AlbumRepository` (to 404 on a missing album) and the `ImageProcessor` component.
- `models/service/` — `readonly` classes implementing `ApiServiceInterface`; business logic and `NotFoundHttpException` on missing records. Validation failures are returned as the model with errors; the base controller converts them to a 422.
- `models/repository/` — implement `ApiRepositoryInterface`; all DB access (queries, batch inserts for seeding). `getAllDP(?SearchCriteria)` builds the `ActiveDataProvider` (default pageSize 20) from the criteria: forced `scope` conditions via `andWhere`, user `filters` via `andFilterWhere` (empty operands skipped), `orderBy`, and page size. Sorting is resolved by the search form, so the provider's own `Sort` is disabled (`'sort' => false`).
- `models/form/` — form requests extending `basic/ApiForm` (a `yii\base\Model`); validate raw request data in the controller before anything reaches the service. Two flavours:
  - **Write forms** (`*CreateForm`/`*UpdateForm` over an abstract per-resource base): validate body params. Create adds `required`; Update is all-optional (partial updates). Only `validatedData()` (attributes actually present in the request) is passed on, so clients can't set server-managed fields (`auth_key`, `access_token`, `password_hash` — clients send plain `password`, hashed in `UserService`). Album ownership is also server-managed: `AlbumForm` deliberately omits `user_id`, so `AlbumsController::actionCreate` forces it to `Yii::$app->user->id` (the album belongs to the authenticated caller) and update can never reassign it. Concrete controllers implement `createForm()`/`updateForm()`.
  - **Search forms** (`*SearchForm` extending `basic/SearchForm`): validate the *query* params of index endpoints (`sort`, `per_page` ≤ 100, plus one attribute per filter) and produce a `SearchCriteria` DTO. Each resource declares `sortableAttributes()` (whitelist — an unknown `sort` field 422s) and `likeAttributes()`/`exactAttributes()` for filtering; a `-` prefix means descending. Concrete controllers implement `searchForm()`. The base `ApiController::actionIndex` runs the shared `handleIndex()` (validate query params → criteria → `service->getAll($criteria)`); `PhotosController` overrides it to pin the album scope via `PhotoService::getByAlbum($albumId, $criteria)`.
- `models/db/` — ActiveRecord models (`User`, `Album`, `Photo`). `Photo::$source` (`'seed'` | `'photo'`) records where a file lives so `Photo::getUrl()` can build the right link; the URL logic itself lives in `components/PhotoUrlBuilder` (single source of truth, used by the model and by tests — never hand-concatenate a photo URL).
- `models/dto/` — readonly DTOs: response shapes with `fromModel()`/`toArray()`, plus `SearchCriteria` (the resolved list-query spec passed from search form → service → repository).

**Photo uploads**: `POST /albums/<id>/photos` is `multipart/form-data` with `title` + `file`. `models/form/PhotoCreateForm` accepts only `jpg, jpeg, png, webp, gif, avif` (extension whitelist; `checkExtensionByMimeType` off) and requires an actual uploaded file. `components/ImageProcessor` (Imagick) converts every upload to **WebP quality 80**, scaled to fit **500×500** preserving aspect ratio (never upscaled); the real content is validated here (a non-image passing the extension check fails and returns 422). It does **not** touch the filesystem itself — persistence is delegated to an injected **`League\Flysystem\FilesystemOperator`** (bound in `config/di.php` to a `LocalFilesystemAdapter` rooted at the `photo_upload_path` param; swap for `AwsS3V3Adapter` to move storage to S3 with no code change — `league/flysystem-aws-s3-v3` is installed). Files are keyed `<albumId>/<random>.webp`. `PhotoService` sets `source = 'photo'`, and cleans up the file on failed validation or on delete. `PhotoUpdateForm` allows changing **only the title** — the album and the stored file are immutable. Seeded photos use `source = 'seed'` and resolve to `web/default-images/`. `config/test.php` overrides the `FilesystemOperator` binding to a `@runtime`-rooted local adapter so test uploads never touch the web root. The web server must be able to write `web/uploads` (`setup.sh` chmods it 777).

**Background jobs (queue)**: slow, retriable side-effects go through a small queue seam — `models/contract/queue/QueueInterface` (`push(JobInterface)`) + `JobInterface` (`handle()`). Two drivers in `components/queue/`: `DbQueue` (default, bound in `config/di.php`) persists a serialized job to the `queue_job` table, and `SyncQueue` runs it in-process (bound in `config/test.php` so tests don't need a worker). The long-running **`worker`** compose service runs `commands/QueueController::actionListen` (`yii queue/listen`) — drains continuously, sleeps only when idle, and stops gracefully on SIGTERM via `pcntl` (baked into the `base` Docker stage); `yii queue/run` drains once for CI/manual use. Jobs (`models/jobs/`) carry only plain data and resolve services from the container at run time (so they serialize cleanly). Current job: `DeleteAlbumDirectoryJob` — `AlbumService` enqueues the on-disk cleanup of a permanently-deleted album's upload directory instead of doing it inline. **yiisoft/yii2-queue is deliberately not used**: its current release caps `symfony/process` at `^7` while this project is on `^8` (PHP 8.5); on a mainstream stack it would back the same `QueueInterface`.

**Transactions & concurrency**: multi-row mutations run inside a DB transaction via the injected `models/contract/service/TransactionRunnerInterface` (`components/DbTransactionRunner` wraps `Yii::$app->db->transaction()`; unit tests pass an inline fake, so services stay DB-free). `RoleService` (create/update/delete/assignRoles) and `UserService::delete` (account + owned albums/photos/files) are wrapped. The RBAC last-role-manager invariant is made concurrency-safe by `RoleRepository::lockManageHolders()` — a `SELECT ... FOR UPDATE` on the `user_role` rows granting `role.manage`, taken inside the transaction before the count check, so concurrent mutations serialize instead of both passing the check.

**Authentication (two-token model)**: access tokens are **stateless**, refresh tokens are **stateful**. The split is deliberate — the access token is checked on every request (so it must be cheap = no DB hit), while the refresh token is checked rarely (once per access-token lifetime), which is exactly where we can afford to reintroduce DB state to gain **revocability**.

- **Access token** — every resource endpoint requires `Authorization: Bearer <token>` (`HttpBearerAuth`, attached after the CORS filter in `ApiControllerTrait` with `except => ['options']` so preflights stay public). It is a stateless HS256 JWT issued/validated by `components/JwtService.php` (app component `jwt`; secret/TTL from `JWT_SECRET`/`JWT_TTL` in `config/di.php` — the secret must be ≥ 32 chars, changing `.env` requires recreating the web container). `User::findIdentityByAccessToken()` resolves the user from the token's `sub` claim; nothing is stored in the DB.
- **Refresh token** — an **opaque** high-entropy string (NOT a JWT), managed by `models/service/RefreshTokenService` over `models/repository/RefreshTokenRepository` + the `refresh_token` table. Only the token's **SHA-256 hash** is stored (a DB leak exposes no usable tokens); the raw value is returned to the client once. Each row belongs to a **family** (`family_id` = one login session / device). Config: TTL from `JWT_REFRESH_TTL` (default 30 days) bound in `config/di.php`. The table has a unique index on `token_hash` (the hot lookup) and a `user_id` FK with `ON DELETE CASCADE`, so a deleted user's tokens vanish — which is why refresh needs no separate "user still exists" check.
  - **Rotation + reuse detection** (`RefreshTokenService::consume()`): a refresh token is single-use — exchanging it revokes it and issues a successor **in the same family**. Presenting an already-revoked token means a copy is loose, so the **whole family is revoked** (forces re-login). Revocation is a soft `revoked_at` timestamp (rows are kept so reuse can be detected).

The **auth flow** lives in `AuthController` (all endpoints public, all rate-limited) through `AuthService` / `AuthServiceInterface`. `AuthService` composes `UserService` (account creation), `RefreshTokenService` (refresh lifecycle) and `JwtService` (access tokens); its private `issueTokens()` returns the `TokenResponse` DTO (`access_token` + `refresh_token` + `token_type` + `expires_in`) for login/register/refresh:
- `POST /auth/login` — `LoginForm` → `AuthService::login()` → `UserRepository::findByEmail()`; bad credentials → 401. Starts a new token family.
- `POST /auth/register` — public self-registration (closes the chicken-and-egg loop: no token needed to get an account). Reuses `UserCreateForm` and delegates creation to `UserService::create()` (password hashing / server-managed fields stay in one place), then logs the user straight in → 201; duplicate email / missing field → 422. `AuthService::register()` returns `User|TokenResponse`.
- `POST /auth/refresh` — `RefreshTokenForm` (`refresh_token` required) → `AuthService::refresh()` → `RefreshTokenService::consume()`, issuing a **fresh pair** in the same family; unknown/expired/revoked token → 401.
- `POST /auth/logout` — revokes the presented token's **family** (log out this device). Best-effort and idempotent: unknown token → still 204, no info leak.
- `POST /auth/logout-all` — revokes **every** family of the token's owner (log out all devices). Both logout endpoints share `RefreshTokenForm` and return **204**.

Email uniqueness is validated at the form level: `UserCreateForm` does a plain unique check, `UserUpdateForm` gets the record id via constructor (`updateForm(int $id)`) and excludes it from the check.

**Authorization (RBAC)**: flat, no role hierarchy or inheritance. A role is a named set of permissions; a user may hold **several** roles and their effective permissions are the **union**. There is no `user` role — a freshly registered account has **no roles** (a "base user"); roles are an upgrade on top of the base abilities. Storage is four own tables (not `yii\rbac`): `permission` (catalog), `role` (`is_system` protects the three seeded roles), `role_permission`, `user_role` (all links cascade on delete). Two migrations seed it: `m260711_000000` (tables) and `m260711_000001` (catalog + `moderator`/`admin`/`super_admin`).

- **Permissions** are code-checked strings, so they live **only in migrations** (their lifecycle is the code's) — there is no create/update/delete for them, only `GET /permissions` (needs `permission.index`) for the role-composition UI. Catalog permissions are all `.any`-scoped "upgrades" (e.g. `album.update.any`, `photo.delete.any`, `user.index.any`, `role.manage`, `role.assign`). Every new migration that adds a permission **must also grant it to `super_admin`** (defined as "holds every permission").
- **`models/service/AccessControlService`** (`AccessControlInterface`, memoized per request, injected everywhere via DI) answers the two questions controllers ask: `can('user.index.any')` (global) and `canOn('album.update', $model)` (per-record). `canOn` allows the action when a role grants the `.any` variant **or** the caller **owns** the subject and the ability is one of the implicit `OWN_ABILITIES` (own album/photo/profile — this is what a base user gets without any role; ownership is resolved polymorphically via `models/contract/OwnableInterface::getOwnerId()`, which `Album`/`Photo`/`User` implement — no `instanceof` switch in the service). Deliberately *not* ownership-granted: `user.view`/`user.delete` and all `*.index`. Because each action checks **only its own** permission, a custom role of just `photo.delete.any` can delete any photo without being able to view it. Denial → `ForbiddenHttpException` (403); `/users/me` and `/users/me/permissions` are the only never-gated endpoints (any authenticated user, so an exotic role can still ask "who am I / what may I do" for the client to build its UI).
- **The three system roles**: `moderator` (see all users; manage all albums but delete only via **soft-delete**; full photo access incl. permanent delete), `admin` (full user CRUD, permanent album delete + restore, `role.index`/`role.assign`), `super_admin` (everything, incl. `role.manage`).
- **Role API** (`RolesController` + `RoleService`/`RoleServiceInterface`): `GET /roles` (`role.index`, name+description), `GET /roles/<id>` (`role.view`, +permissions), and create/update/delete (`role.manage`, super-admin only) composing a role from catalog permissions (unknown name → 422; system roles can be re-composed but not renamed/deleted → 422/409). Assignment is a child resource of users: `GET|PUT /users/<id>/roles` (`role.assign`; PUT is a full replacement, empty array revokes all). Two safety rules in `RoleService`: **anti-escalation** (a holder of `role.assign` without `role.manage` cannot grant/revoke any role carrying `role.manage`/`role.assign` — so an admin can't mint or demote admins) and the **last-role-manager invariant** (no operation — role delete/re-compose, assignment change, user delete — may leave the system with zero holders of `role.manage`; violation → 409). The same guards protect user mutation: an admin can't update/delete a `role.manage` holder's account, and the last super admin can't delete themselves.
- **Pseudo-deletion of albums** (`album.is_deleted` + optional `album.delete_reason`): `DELETE /albums/<id>` has one route, outcome by permission — permanent for whoever may delete outright (the owner, or `album.delete.any`; permanent wins if a role has both), soft (flag + optional `reason`, idempotent) for `album.soft-delete.any` only. Soft-deleted albums are hidden from every listing by default (`AlbumSearchForm`'s `is_deleted` filter / `?is_deleted=1` surfaces the review queue for `album.index.any`) and are a 404 for their owner until an admin `POST /albums/<id>/restore` (`album.restore`). The review audience (`album.view.any`/`photo.view.any`) still sees them.
- **First super admin**: everything RBAC-driven needs an existing `role.manage` holder, so bootstrap from the console — `make rbac-assign role=super_admin email=<email>` (`RbacController::actionAssign`, idempotent). Registration/user-creation assign **no** role.

**Login rate limiting (brute-force protection)**: `components/RateLimiter.php` is a cache-backed `ActionFilter` attached in `AuthController::behaviors()` that throttles the auth endpoints per client IP, with an **independent counter per action** (the cache key includes `$action->getUniqueId()`, so login/register/refresh/logout don't share a budget) — max attempts / window configured once in `config/di.php` from `LOGIN_RATE_LIMIT_ATTEMPTS`/`LOGIN_RATE_LIMIT_WINDOW` env vars (defaults 5 / 60s). Every non-OPTIONS request increments the counter and refreshes the window; exceeding the limit → 429 with a `Retry-After` header; a response < 400 (a success) resets the counter, while failures (401 thrown, 422 set) do not. It uses the `cache` app component (FileCache; `config/test.php` points it at `@runtime/test-cache`, and `BaseCest::_before` flushes it so counters don't leak between tests).

**Health check**: `GET /health` (`HealthController`, public, unauthenticated, no rate limiting — plain `yii\rest\Controller` + `ApiControllerTrait` like `AuthController`) calls `HealthService::check()` (`HealthServiceInterface`), which runs `SELECT 1` against the injected `yii\db\Connection` and returns a `HealthCheckResult` DTO (`{"status": "ok"|"error", "checks": {"database": ...}}`). Since `db` is an app *component* rather than a container-managed class, it can't be wired with `Instance::of()` (that only resolves classes/definitions known to the container) — `config/di.php` binds `HealthService::class` to a closure that pulls `Yii::$app->db` directly. 200 when healthy, 503 otherwise.

**API documentation (OpenAPI/Swagger)**: `config/openapi.yaml` (OpenAPI 3.0) is the **single source of truth** for the API's request/response shapes and RBAC gates — keep it in sync whenever an endpoint changes (there is no generated-from-code spec; the trade-off for zero code clutter is that the YAML is updated by hand, and CI could lint it). `DocsController` serves it publicly at `GET /docs` (Swagger UI, HTML loaded from a pinned `swagger-ui-dist` CDN) and `GET /docs/openapi.yaml` (raw spec). It extends plain `yii\web\Controller` (not the `ApiController`/REST stack) and sets `Response::FORMAT_RAW` per action, so the docs deliberately bypass the JSON envelope and JWT auth. Routes are the first two explicit rules in `config/url_rules.php`. This replaced the old hand-written `docs/api/API.md`.

**Unified response format**: every response is `{"success": bool, "data": ..., "code": int}`. This is enforced in two places — `components/ApiSerializer.php` (wraps normal REST responses, converts ≥400 statuses to the error shape) and `components/JsonErrorHandler.php` (renders uncaught exceptions in the same shape, with file/line/trace only when `YII_DEBUG`). New endpoints get this automatically; don't hand-build response envelopes.

**Routing**: the shared rule table lives in `config/url_rules.php` (required by **both** `web.php` and `test.php`, which keep only their own `urlManager` flags like `enableStrictParsing`; `test.php` also overrides `ImageProcessor::$uploadPath` to `@runtime` so test uploads never touch the web root). A `yii\rest\UrlRule` with `pluralize => false` covers `users`, `albums`, `roles`; a `photos` `UrlRule` with `except => ['index', 'create']` leaves only `/photos/<id>` (view/update/delete) — no flat collection (`GET /photos` → 405). Everything else is an explicit rule declared **before** those UrlRules so it wins: photos nested under their album (`GET|POST albums/<albumId>/photos`), the current-user endpoints (`GET users/me`, `GET users/me/permissions` → `users/me`, `users/me-permissions`), role assignment as a child of users (`GET|PUT users/<id>/roles`), `albums/my`, `POST albums/<id>/restore`, and the read-only `GET permissions`. Adding a resource means: migration, AR model, repository + service (with contracts), create/update/search form requests, a controller extending `ApiController` that implements `accessResource()`, seeding its permissions in a migration (and granting them to `super_admin`), and adding the controller to `config/url_rules.php`.

**Console commands** live in `commands/` (namespace `app\commands`), extend `basic/BasicConsoleController` (prints elapsed execution time after each action), and use the same constructor DI pattern as web controllers. `config/console.php` maps `migrate`, `migrate-test` (same `migrations/` path, different DB component), and `migration-creator`; other controllers are auto-discovered from the namespace (`SeederController` → `seeder`, `RefreshTokenController` → `refresh-token`, `RbacController` → `rbac`). `rbac/assign <role> <email>` (`make rbac-assign role=… email=…`) appoints the first super admin (idempotent) — see **Authorization (RBAC)**. `refresh-token/prune` (`make refresh-token-prune`) hard-deletes fully-expired `refresh_token` rows — expired tokens can't be exchanged and aren't needed for reuse detection, so it's meant to run on a cron to keep the table bounded (still-valid revoked rows are kept).

## Testing Conventions

- Functional tests (`tests/functional/`) extend `BaseCest`, hit real endpoints via REST module, and insert fixtures with `$this->insertRecord('table', [...])` against the test DB. `BaseCest::_before` truncates tables (and deletes non-system roles, keeping the migration-seeded catalog/roles as reference data), then creates an auth user (`$this->authUserId`, email `auth.user@example.com`) with the **`super_admin`** role and sets a Bearer token for every request — so the shared CRUD tests aren't blocked by RBAC; user-list pagination totals include this user. RBAC helpers: `actingAsUserWithRole($I, $role, $overrides)` inserts a fresh user (pass `null` for a roleless base user) and authenticates as them; `assignRole()`/`insertRole($name, $permissions)`/`roleId()` manage roles; `sendPutJson($I, $url, $body)` PUTs a JSON body (needed when the body carries a possibly-empty array — form encoding drops empty arrays, so `config/test.php` also registers the JSON parser like production). Call `$I->deleteHeader('Authorization')` to test unauthenticated behavior (see `AuthCest`).
- Unit tests (`tests/unit/`) test services in isolation with PHPUnit mocks of the repositories (and of `ImageProcessor` for `PhotoService`) — no DB (except rules that query during `validate()`, e.g. `User`'s `unique` email and `Photo`'s `album_id` `exist`).
- `PhotosCest` uploads with `$I->sendPost($url, ['title' => ...], ['file' => $path])` and generates image fixtures on the fly with Imagick; it asserts the stored file is WebP and correctly resized. Photo fixtures inserted directly need a `source` value (`'seed'`/`'photo'`).
- `config/test.php` uses strict URL parsing and disables CSRF; test DB config is `config/test_db.php`.

## CI/CD & Static Analysis

**CI** — `.github/workflows/ci.yml` runs on every push and pull request: `composer install`, then three gates — PHP CS Fixer (`--dry-run`), PHPStan, and the full Codeception suite against a MySQL 8 service. It runs natively on the runner (PHP 8.5 + Imagick via `shivammathur/setup-php`), not through Docker; DB/JWT/rate-limit config is passed as workflow `env:` (the same vars the app reads via `getenv()`), and only the test DB is migrated (`migrate-test/up`). Keep the CI steps in sync with the `Makefile` targets (`cs-check`, `stan`, `test`) — they must stay runnable both ways.

**CD** — `.github/workflows/cd.yml` is chained to CI via `workflow_run` (triggers only after the **CI** workflow completes on `master`) and both its jobs guard on `github.event.workflow_run.conclusion == 'success'`, so a red CI never deploys (and its badge stays neutral rather than red). It has two jobs: `build-image` builds the `prod` stage of the multi-stage `Dockerfile` (`target: prod` — `composer install --no-dev --no-scripts` + baked app code, on top of the shared `base` stage) with buildx + GHA layer cache, then smoke-tests it (`php --version` inside the image); `deploy` runs through a `production` GitHub Environment so it shows in the repo's Environments/Deployments tab, but the actual release is **simulated** (echoed steps) — this sample deliberately provisions no real server. The image is built with `push: false`/`load: true` (never pushed to a registry). The `prod` stage uses `--no-scripts` so Yii's `postInstall` (cookie-key generation) is skipped — `COOKIE_VALIDATION_KEY` comes from env at runtime instead — and only `yiisoft/yii2-composer` is in `allow-plugins`. Dev and prod share the `base` stage, so there is no runtime setup to keep in sync between compose and the image.

PHPStan config is `phpstan.neon.dist` (level 5, scans `commands`/`components`/`config`/`controllers`/`models`). Two non-obvious settings make it green without false positives: `bootstrapFiles` + `scanFiles` both point at `vendor/yiisoft/yii2/Yii.php` so the global `Yii` class (a generic subclass of `BaseYii`) is both executed for its constants and reflected for its inherited static methods (`Yii::getAlias()` etc.); and `dynamicConstantNames` lists `YII_DEBUG`/`YII_ENV*` because `BaseYii` defines them as `false`/`'prod'` defaults, so without this PHPStan reports every `if (YII_DEBUG)` branch as dead code. PHPStan needs `--memory-limit=512M` (the web container's PHP caps at 128M). Fix real type errors rather than lowering the level or adding ignores.
